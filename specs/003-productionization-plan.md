# 003 — 고찌봄 실서비스화 계획서 (Productionization Plan)

## 0. 요약 (TL;DR)
지금 배포된 앱(https://gojjibom.web.app)은 **와이어프레임 충실도 + 목(mock) 데이터**의 정적 SPA다.
실서비스가 되려면 다섯 축이 필요하다: **(1) 백엔드 API·DB, (2) 인증·가족방·기기 페어링,
(3) 복약 스케줄·알림·미응답 에스컬레이션, (4) 복약 확인·실시간 모니터링, (5) 민감 건강정보 보안·개인정보 준수.**
아래는 FE(이 레포) 중심 작업 + BE/인프라 파트너 트랙 + 단계별 로드맵 + 결정 필요 항목이다.
"약을 추천하지 않고 약국 등록 정보만 안내"하는 비(非)의료기기 포지셔닝은 그대로 유지한다.

## 1. 현재 상태 (As-is)
- FE: Vite + React 19 + TS + Tailwind + react-router. 모든 화면이 `src/lib/mock.ts` 하드코드 데이터로 렌더.
- 데이터 흐름 없음: API 호출/인증/알림 전무. QR·연결코드·전화·음성은 모두 시늉(stub) 또는 화면 전환뿐.
- 셸: 데모용 아이폰 프레임 + 상단 역할 토글(어르신/보호자·복지사) — 실제 앱 UI가 아니라 프리뷰 장치.
- 인프라: Firebase Hosting(project `gojjibom`) + GitHub Actions 자동배포. BE 타깃 `VITE_API_BASE_URL`
  (`https://api.stdiodh.xyz/api`)만 정의돼 있고 실제 연동은 없음. `VITE_DEMO_MODE=true`로 빌드됨.
- 결론: "보여주기용 프로토타입" 단계.

## 2. 목표 (To-be)
운영 가능한 서비스. 보호자가 가족방을 만들고 어르신 기기를 페어링, 약국 등록 복약정보로 스케줄을 구성,
정해진 시각에 알림 → 어르신 원터치/음성 복약확인 → 미응답 시 재알림·보호자/복지사 에스컬레이션,
보호자·복지사는 실시간으로 상태·개수·타임라인을 모니터링. 민감정보는 동의·암호화·접근통제로 보호.

## 3. 갭 분석 (무엇이 없나)

| 영역 | 현재 | 실서비스에 필요 |
| --- | --- | --- |
| 인증/세션 | 없음 | 보호자 휴대폰 OTP 로그인, 어르신 기기 토큰(무로그인) |
| 가족방 | 목 | 생성/초대(코드·QR)/구성원 권한(RBAC) |
| 복약 데이터 | 하드코드 | 약국 등록 처방·봉지·시간, 변경 이력 |
| 알림 | 없음 | 예약 알림 + 재알림 + 에스컬레이션(푸시/SMS/전화) |
| 복약 확인 | 화면 전환만 | 실제 이벤트 기록, 확인 수단(버튼/음성), TTS 안내 |
| 모니터링 | 목 | 실시간 상태, 개수 추정·소진 예측, 타임라인 |
| 보안/개인정보 | 없음 | 암호화·접근통제·동의·보관/파기·감사로그 |
| 관측성 | 없음 | 로깅·모니터링·알림 실패 추적·에러 리포팅 |

## 4. 타깃 아키텍처
- **FE (SPA/PWA)** on Firebase Hosting ↔ **BE REST/JSON API** (`api.stdiodh.xyz/api`) ↔ **DB**.
- **알림 파이프라인**: 서버 스케줄러(cron/queue) → 웹푸시(FCM) + SMS/전화(외부 제공자). 미응답 시 단계별 에스컬레이션.
- **어르신 클라이언트**: 웹 PWA 가정(전용기기/키오스크 모드 옵션). 기기 페어링 토큰으로 무로그인 사용.
- **데이터 출처**: 약국 등록 복약정보(연동 API 또는 관리자 수기 입력) — §12 결정 필요.
- FE는 BE의 API 계약(OpenAPI)에만 의존. 이 레포는 계약이 정해지면 mock을 실제 호출로 교체.

## 5. 데이터 모델 (핵심 엔티티 초안)
- `Account` (보호자/복지사, role), `Device` (어르신 기기, 페어링 토큰)
- `CareCircle` (가족방) — members[Account+role], seniorProfile
- `SeniorProfile` (어르신)
- `Medication` (약, 약국 등록), `Packet` (봉지)
- `DoseSchedule` (시각/식전·식후/봉지/요일)
- `DoseEvent` (예정→알림→확인/미응답), `Confirmation` (수단: voice|button, 시각)
- `PillInventory` (남은 개수, 소진 예측, 재처방 D-day)
- `NotificationJob` (예약/재알림/에스컬레이션 단계), `EscalationPolicy`
- `Consent` (동의), `AuditLog`
- 이 계약에 맞춰 `src/lib/types.ts`를 정렬/확장한다(지금 타입이 좋은 출발점).

## 6. 기능별 실구현 (feature → real)
각 항목: **FE 작업 / BE 필요 / 비고**.
1. **인증·가족방·페어링** — FE: 로그인/코드·QR 화면 실동작, route guard, 토큰 처리. BE: OTP, 기기 토큰, 가족방 CRUD/초대.
2. **복약 스케줄** — FE: 오늘/다음 약을 API로. BE: 약국 데이터 or 입력 백오피스, 스케줄 계산.
3. **알림·에스컬레이션(난이도 최상)** — FE: 웹푸시 구독, 알림 클릭→확인 화면. BE: 스케줄러, 재알림, 미응답 판정, 보호자/복지사 통지, 타임존/DST.
4. **복약 확인** — FE: 원터치 확인 이벤트 전송, `SpeechSynthesis` ko-KR 음성(자동재생 정책→버튼 트리거), 확인 수단 기록. BE: 이벤트 저장/집계.
5. **모니터링 대시보드** — FE: 실데이터 + 폴링/웹소켓/푸시 갱신, 개수·타임라인. BE: 상태 집계 API.
6. **접근성(어르신)** — FE: 큰 글씨·고대비 토글, 포커스/키보드, 스크린리더 라벨, 단순 인터랙션(WCAG AA 지향).

## 7. 비기능 요구사항
- **보안/개인정보(민감 건강정보)**: 전 구간 TLS, 저장 암호화, 최소 수집, RBAC, 동의/약관, 보관·파기 정책,
  로그 마스킹. 국내 개인정보보호법·민감정보 별도 동의 검토(법률 자문 필요). 비의료기기 포지셔닝 유지.
- **알림 신뢰성**: 타임존/DST 정확성, 중복·누락 방지, 재시도, 아이들포턴시. 웹푸시 한계(특히 iOS) 대비 SMS/전화 백업.
- **성능/접근성/관측성/오프라인(PWA)**: 초기 로드 최적화, 에러 바운더리, Sentry 등, 오프라인 표시.

## 8. FE 트랙 (이 레포에서 우선 할 일)
- **API 계층 도입**: `src/lib/mock.ts` → 어댑터 뒤로 숨기고 실제 `fetch`. 서버 상태관리 `@tanstack/react-query` 권장.
  현재 없는 **로딩/에러/빈 상태**를 전 화면에 도입.
- **환경 정리**: 실서비스 빌드는 `VITE_DEMO_MODE=false`(=실 API), 데모/발표용만 `true` 폴백.
- **인증/세션**: 토큰은 민감정보 → `localStorage` 금지, httpOnly 쿠키 등 안전한 방식. role별 route guard, 어르신 기기 토큰.
- **셸/라우팅**: 데모용 아이폰 프레임 + 역할 토글은 실환경에선 제거/분기(어르신 앱 vs 보호자 앱). 역할은 인증으로 결정.
- **입력/폼**: OTP·연결코드 실제 검증, QR 실제 스캔(BarcodeDetector/jsQR + 카메라 권한).
- **PWA**: manifest + service worker + 웹푸시 구독 + 설치. **음성**: `SpeechSynthesis` ko-KR.
- **테스트/관측성**: 라우팅·상태·에러 케이스 유닛+통합(MSW로 API 목킹), 에러 리포팅.

## 9. BE / 인프라 트랙 (경계 — 별도 BE 레포/팀)
- **API 계약(OpenAPI)** 확정: §5 엔티티 CRUD + 이벤트/확인/알림 상태.
- 인증(휴대폰 OTP + 기기 페어링 토큰 + JWT/세션), 스케줄러+큐(알림/재알림/에스컬레이션).
- 푸시(FCM)/SMS/전화 외부 연동, 약국 데이터 연동 또는 관리자 입력 백오피스.
- AWS 배포·모니터링, 개인정보 저장·백업·파기 정책.

## 10. 로드맵 (단계별 마일스톤)
- **M0 — 기반**: API 계약 합의 + 보호자 로그인 + 가족방/초대 + 어르신 기기 페어링(실동작). FE: API 계층+guard.
- **M1 — 핵심 복약 루프**: 스케줄·오늘 복약·복약확인 실연동, 대시보드 실데이터. (해커톤 MVP 지점)
- **M2 — 알림·에스컬레이션**: 웹푸시 + 재알림 + 미응답 시 보호자/복지사 통지, 타임라인.
- **M3 — 약 개수/재처방 · 음성 안내 · 접근성 · PWA**.
- **M4 — 보안/개인정보/동의 · 관측성 · 하드닝 · 베타 배포**.

## 12. 리스크 & 결정 필요 (오픈 퀘스천)

### 확정된 결정 (2026-07-02)
- **API**: `https://api.stdiodh.xyz/swagger-ui/index.html` 에 추후 명세 예정. FE는 그 계약을 따른다.
  **현재는 mock 데이터 유지**(`VITE_DEMO_MODE=true`) — API가 준비되면 `src/lib/api.ts`의 엔드포인트를 채우고
  `VITE_DEMO_MODE=false`로 전환하면 실제 호출로 바뀐다(demo-mode fallback 어댑터, §8).
- **UX**: "실질적 서비스처럼" — 와이어프레임 프레임(아이폰 베젤·상태바)과 상단 역할 토글 제거, 반응형
  (모바일 풀스크린 / PC 중앙 앱 카드) + PWA 메타(manifest·theme-color·아이콘). 역할 진입은 온보딩에서.

### 남은 오픈 퀘스천
1. **약국 복약정보 데이터 소스** — 연동 API가 있나, 관리자 수기 입력인가? (전체 설계의 핵심)
2. **어르신 클라이언트 형태** — 웹 PWA인가, 전용 앱/기기인가? 웹푸시 신뢰성 한계(iOS) → SMS/전화 백업 필수?
3. **SMS/전화 제공자** — Twilio vs 국내(알리고/NHN 등), 비용·규제.
4. **개인정보(민감 건강정보) 동의·법적 검토** 주체와 범위.
5. **BE 소유권** — `api.stdiodh.xyz`(및 Tiki-Taka-BE 레포) 팀/계약 소유. FE는 계약만 맞추면 됨.
6. **인증 토큰 저장 방식**(보안) 및 세션 정책.
7. **"전화하기"** — 클라이언트 `tel:`로 충분한가, 서버 경유 콜/기록이 필요한가.
